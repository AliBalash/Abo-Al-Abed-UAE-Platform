# Architecture Notes

## Monorepo layout

- `apps/api`: NestJS modular monolith, Prisma/PostgreSQL schema, order state machine, cashier/kitchen/admin endpoints.
- `apps/web`: Next.js role-based internal panel for `cashier`, `kitchen`, and `ops/admin`.
- `apps/ios`: SwiftUI iPhone app scaffold generated by XcodeGen.
- `packages/contracts`: shared zod schemas and domain enums for the JS stack.
- `packages/ui`: shared web design tokens.

## Core runtime choices

- Backend: NestJS + Prisma + PostgreSQL/PostGIS-ready schema + WebSocket gateway.
- Web: Next.js App Router, custom CSS, no design-system bloat.
- iOS: SwiftUI, async/await, mock/live hybrid data client.
- Infra: Postgres with PostGIS extension and Redis via `docker-compose.yml`.

## Ordering lifecycle

1. Customer authenticates with email/password.
2. Customer selects a saved map-backed address.
3. Customer browses the shared menu and customizes products.
4. Backend recommends the nearest active branch.
5. Order is created in `awaiting_payment` with `orderCode` and `pickupToken`.
6. Cashier confirms manual payment in the branch web panel.
7. Kitchen progresses order through `paid` → `in_preparation` → `ready_for_pickup` → `picked_up`.
8. Notifications and websocket events are emitted on each operational milestone.

## Known implementation boundaries

- The Prisma schema is production-oriented, but actual migrations and runtime validation still need a successful dependency install and database connection.
- The iOS app includes a live client shell plus mock fallback because this environment does not have full Xcode configured.
- Google Sites / Linktree are treated as source references only; production assets still need re-hosting to S3-compatible storage.
